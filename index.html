    CHAPTER 1

    1장. Skill 이해하기

    Codex에게 매번 다시 설명하던 일을 저장해 두는 방법

    🎯 이 장에서 배울 것
    • Skill의 정확한 의미
    • 프롬프트, 플러그인, 서브에이전트와의 차이
    • Codex가 Skill을 읽고 사용하는 방식
    • Skill이 결국 한 개의 'md 문서'라는 사실

    1.1 Skill의 한 문장 정의

    정의. Skill은 Codex가 특정 업무를 할 때 참고하는 재사용 가능한 작업 지침 폴더다. Skill 안에는 반드시 SKILL.md 파일이 있고, 필요하면 스크립트·참고문서·템플릿 같은 리소스가 함께 들어간다.

    ★ 핵심 문장

    Skill은 '이번 한 번의 지시'가 아니라 '앞으로도 반복해서 쓸 업무 방식'이다.

    예를 들어 다음과 같은 일은 Skill로 만들기 좋다.

    • 매주 같은 형식으로 데이터를 요약한다.
    • 항상 같은 회사 양식으로 보고서를 작성한다.
    • PDF, 엑셀, 음성 파일을 정해진 방식으로 처리한다.
    • 프로젝트를 시작할 때 요구사항을 늘 같은 기준으로 정리한다.

    1.2 Skill과 다른 도구의 차이

    구분무엇인가언제 쓰는가기억할 점
    프롬프트현재 대화에서 쓰는 지시문한 번만 하면 되는 작업대화가 끝나면 다시 설명해야 한다
    Skill재사용 가능한 업무 지침과 리소스반복되는 업무, 표준 절차가 있는 작업다음에도 같은 방식으로 시작할 수 있다
    플러그인 / 커넥터외부 서비스와 연결하는 기능메일, 캘린더, GitHub 같은 서비스 작업서비스 연결이지 업무 절차 자체는 아니다
    서브에이전트작업을 나눠 병렬로 처리하는 실행 단위조사, 구현, 검증을 분리할 때Skill은 기준을 주고, 서브에이전트는 일을 나눈다

    1.3 Codex가 Skill을 읽는 순서

    1. Codex는 사용 가능한 Skill의 이름과 description(설명)을 먼저 본다.
    2. 사용자의 요청이 어떤 Skill과 맞으면 그 Skill이 발동된다.
    3. Skill이 발동된 뒤에야 SKILL.md 본문을 읽는다.
    4. 본문에 적힌 참고문서·스크립트·템플릿이 필요하면 그때 해당 리소스를 사용한다.
    구성 요소언제 보이는가역할
    name / description항상 얇게 보임이 Skill을 쓸지 말지 판단한다
    SKILL.md 본문Skill 발동 후실제 작업 순서와 기준을 알려준다
    references필요할 때긴 정책, 예시, 스키마, 설명 자료를 담는다
    scripts필요할 때정확히 반복해야 하는 작업을 실행한다
    assets필요할 때템플릿, 이미지, 샘플 파일 같은 재료를 담는다

    1.4 실습 — Skill의 정체를 직접 확인하기

    Skill은 마법이 아니라 '글로 적힌 지침서'다. 이 사실을 눈으로 직접 확인해 본다.

    ⌨ 따라 입력해보기

    현재 사용할 수 있는 Skill 중 하나를 골라, 그 Skill의 SKILL.md 파일이 어디에 있는지 실제 경로를 알려줘. 그리고 그 파일 내용을 열어서 한국어로 번역해 보여줘.

    👁 관찰 포인트

    SKILL.md를 열어 보면 특별한 코드가 아니라, 사람이 읽을 수 있는 설명과 절차가 적힌 문서임을 알 수 있다. Skill이 하는 일은 결국 'Codex가 이 문서를 읽고 그대로 따르는 것'뿐이다.

    CHAPTER 2

    2장. 기본 Skill 사용하기

    아무것도 설치하지 않아도 쓸 수 있는 Skill

    🎯 이 장에서 배울 것
    • Codex에 기본으로 들어있는 Skill
    • Skill을 직접 지명해서 사용하는 법
    • 이미지 생성 Skill을 직접 체험하기
    • SKILL.md를 읽고 'Skill = 가이드 문서'임을 이해하기

    2.1 기본으로 들어있는 Skill

    Codex App은 아무것도 따로 설치하지 않아도 바로 쓸 수 있는 기본 Skill을 갖고 있다. 문서 작성, 표 처리, PDF 다루기, 이미지 생성처럼 자주 쓰는 작업이 미리 준비되어 있다.

    ⚠ 주의

    기본 Skill의 정확한 이름과 개수는 Codex App 버전에 따라 다를 수 있다.

    ⌨ 따라 입력해보기

    지금 내가 따로 설치하지 않아도 바로 쓸 수 있는 기본 Skill에는 어떤 것들이 있어? 각각 한 줄로 설명해줘.

    2.2 Skill을 직접 지명하기

    처음 배울 때는 Codex가 알아서 Skill을 고르기를 기다리기보다, 사용할 Skill 이름을 직접 말하는 편이 좋다. Skill 이름 앞에 $를 붙이거나, 자연어로 Skill 이름을 말하면 된다.

    ⌨ 따라 입력해보기

    $pdf Skill을 사용해서 이 PDF의 핵심 내용을 요약해줘.

    $xlsx Skill을 사용해서 이 CSV를 표로 정리하고 이상값을 찾아줘.

    2.3 실습 — 이미지 생성 Skill 써보기

    기본 Skill 하나(이미지 생성)를 직접 지명해 결과물을 만들어 본다. 원하는 그림을 글로 자세히 묘사할수록 결과가 좋아진다.

    ⌨ 따라 입력해보기 · 이미지 만들기

    imagegen skill을 사용해서, 따뜻한 아침 햇살이 들어오는 책상 위에 노트북, 커피잔, 노트가 놓여 있는 사실적인 사진을 만들어줘. 글자나 로고는 넣지 마.

    💡 팁

    결과가 마음에 들지 않으면 한 가지씩만 바꿔 다시 요청한다. 예: '조금 더 밝게', '위에서 내려다본 각도로', '커피잔을 왼쪽으로'.

    2.4 실습 — Skill의 SKILL.md 읽어보기

    방금 사용한 Skill이 어떻게 만들어져 있는지 들여다본다.

    ⌨ 따라 입력해보기

    방금 사용한 imagegen Skill의 SKILL.md 파일을 열어서, 어떤 내용이 적혀 있는지 한국어로 번역해 설명해줘.

    ★ 핵심 문장

    Skill은 특별한 프로그램이 아니라 '가이드 문서'다. SKILL.md에 적힌 지침을 Codex가 읽고 그대로 따르는 것이다. 이 한 문장이 이번 수업 전체의 핵심이다.

    CHAPTER 3

    3장. Skill 설치하기

    다른 사람이 만든 Skill을 내 Codex로 가져오기

    🎯 이 장에서 배울 것
    • '설치'가 실제로 의미하는 것
    • GitHub에 있는 Skill을 설치하는 흐름
    • 설치 후 재시작과 확인
    • 설치한 Skill을 실제 작업에 활용하기

    3.1 설치란 무엇인가

    Skill 설치는 Codex가 읽을 수 있는 위치에 Skill 폴더를 놓는 일이다. 설치된 Skill은 Codex가 다시 시작될 때 목록에 반영된다.

    ⚠ 주의

    새 Skill을 설치한 뒤에는 Codex App을 재시작해야 한다.

    3.2 GitHub에서 Skill 설치하기

    다른 사람이 만든 Skill은 보통 GitHub에 공개되어 있다. 주소를 채팅창에 붙여 넣고 설치를 요청하면 된다.

    1. 웹 브라우저(구글 등)에서 원하는 Skill을 검색한다. 예: 'taste-skill'
    2. Skill이 있는 GitHub 저장소 주소를 복사한다.
    3. Codex 채팅창에 주소와 함께 설치를 요청한다.
    4. Skill 폴더 안에 새 폴더가 생겼는지 확인한다.
    5. Codex App을 재시작한다.
    ⌨ 따라 입력해보기 · 이렇게 입력

    https://github.com/Leonxlnx/taste-skill 설치해줘

    ✅ 확인

    설치가 끝나면 Skill 폴더 안에 taste-skill 폴더가 새로 생긴 것을 확인한다. 그 뒤 Codex App을 재시작해야 새 Skill이 목록에 안정적으로 반영된다.

    3.3 실습 — 설치한 Skill 활용하기

    방금 설치한 taste-skill로 결과물을 만들고, 그 결과를 다른 작업에 이어 붙여 본다. 아래 세 가지를 순서대로 요청한다.

    ⌨ 따라 입력해보기 · 결과 1 — 브랜드 킷

    taste-skill을 이용해서 애플(Apple) 브랜드 킷을 만들어줘.

    ⌨ 따라 입력해보기 · 결과 2 — 날씨 대문

    간단한 대문 페이지를 HTML로 만들어줘. 주제는 '오늘의 날씨'.

    ⌨ 따라 입력해보기 · 활용 사례 — 둘을 합치기

    결과2(날씨 대문)에 결과1(애플 브랜드 킷)의 디자인을 적용해줘.

    👁 관찰 포인트

    하나의 Skill이 만든 결과(브랜드 킷)를 다른 작업(날씨 대문)에 이어 붙였다. Skill의 산출물은 그 자리에서 끝나지 않고 다음 작업의 재료가 된다 — 이것이 5장 '연결'의 출발점이다.

    3.4 설치 후 확인하기

    • Codex App을 재시작했는가?
    • Skill 이름을 직접 말했을 때 Codex가 해당 Skill을 사용한다고 응답하는가?
    CHAPTER 4

    4장. Skill 만들기

    내가 반복하는 동작을 Skill로 굳히기

    🎯 이 장에서 배울 것
    • 어떤 업무를 Skill로 만들면 좋은지 판단하기
    • Skill 폴더 구조 이해하기
    • 좋은 description 쓰는 법
    • 내 동작을 직접 Skill로 저장해보기

    4.1 Skill로 만들 만한 업무 고르기

    질문Skill로 만들기 좋음Skill보다 프롬프트가 나음
    반복성비슷한 일을 계속 한다한 번만 한다
    절차성항상 따라야 할 순서가 있다매번 완전히 다르다
    자료성정책, 양식, 예시, 스키마가 있다참고 자료가 거의 없다
    오류 비용빠뜨리면 품질 문제가 생긴다틀려도 쉽게 다시 하면 된다

    4.2 Skill 폴더 구조

    my-skill/ ├─ SKILL.md (필수) Skill 설명과 사용 지침 ├─ scripts/ (선택) 반복 실행 코드 ├─ references/ (선택) 긴 참고문서 └─ assets/ (선택) 템플릿과 샘플 파일

    SKILL.md 하나만 있어도 Skill은 작동한다. 나머지 폴더는 필요할 때만 더한다.

    4.3 좋은 description 쓰기

    description은 Codex가 이 Skill을 쓸지 말지 판단하는 가장 중요한 문장이다. '무엇을 하는 Skill인지'와 '언제 사용하는지'가 모두 들어가야 한다.

    나쁜 description문제좋은 description
    보고서를 잘 써주는 Skill범위가 넓고 입력이 불분명하다회사 표준 월간 보고서 초안을 작성·검수할 때 사용. 입력은 KPI 엑셀과 전월 보고서.
    데이터 분석 Skill어떤 데이터인지 모른다VOC CSV를 정규화하고 중복 제거·토픽 분류·주간 통계를 만들 때 사용.

    4.4 실습 — Playwright Skill로 나만의 Skill 만들기

    Playwright Skill은 평소 마우스로 하던 웹 동작 — 사이트 접속, 페이지 이동, 버튼 클릭 — 을 글로 적어 Codex에게 시킬 수 있게 해준다. 그 동작을 Skill로 저장하면, 다음부터는 한 줄 명령으로 같은 작업을 반복할 수 있다.

    네이버에 접속해 카페로 이동하는 간단한 동작을, 재사용 가능한 Skill로 만들어 본다.

    ⌨ 따라 입력해보기 · 1단계 — 동작 시켜보기

    Playwright Skill을 사용해서 네이버(naver.com)에 접속한 다음, 상단 메뉴에서 '카페'로 이동해줘. 어떤 화면이 보이는지 알려줘.

    ⌨ 따라 입력해보기 · 2단계 — 동작을 Skill로 저장하기

    방금 한 동작 — 네이버 접속 후 카페로 이동 — 을 '네이버-카페-열기'라는 Skill로 저장해줘.

    ⌨ 따라 입력해보기 · 3단계 — 만든 Skill 확인하고 다시 써보기

    방금 만든 '네이버-카페-열기' Skill의 SKILL.md를 열어서 보여줘. 그리고 그 Skill을 사용해서 같은 동작을 다시 한 번 실행해줘.

    ★ 핵심 문장

    내가 한 동작이 글(SKILL.md)로 저장되었다. Skill 만들기는 거창한 코딩이 아니라 '내 동작을 글로 받아 적게 하는 일'이다.

    ⚠ 주의

    웹사이트는 화면 구성이 바뀔 수 있다. 라이브 사이트를 다루는 Skill은 가끔 멈출 수 있으므로, 동작이 단순하고 안정적인 흐름을 고르는 것이 좋다.

    CHAPTER 5

    5장. Skill 연결하기

    여러 단계를 하나의 작업 흐름으로 잇기

    🎯 이 장에서 배울 것
    • 단계를 따로 시키는 것과 한 번에 잇는 것의 차이
    • 큰 일을 작은 단계로 나누는 법
    • 앞 단계의 결과를 다음 단계의 재료로 잇는 법
    • 여러 단계를 한 번의 요청으로 묶는 문장 쓰기

    5.1 '연결'이란 무엇인가

    지금까지는 Skill을 하나씩 따로 썼다. 문서 요약 따로, 이미지 생성 따로. '연결'은 여러 단계를 하나의 작업 흐름으로 잇는 것이다 — 앞 단계의 결과가 다음 단계의 재료가 되도록.

    같은 일을 두 가지 방식으로 시켜 보면 차이가 분명하다.

    방식어떻게 진행되는가
    연결하지 않음'데이터 모아줘' → (결과 확인) → '분석해줘' → (결과 확인) → '보고서로 만들어줘'. 매 단계마다 사람이 끼어들어 다음을 시킨다.
    연결함'데이터를 모아서, 분석하고, 보고서로 만들어줘.' 한 번의 요청으로 단계가 이어진다.
    ★ 핵심 문장

    연결은 '여러 번 나눠 시킬 일을 한 번에 시키는 것'이다. 이것만으로도 손이 크게 준다. 다만 연결은 아직 '사람이 직접 요청'한다. 사람 없이 스스로 도는 '자동화'는 6장에서 한 걸음 더 나아가 만든다.

    5.2 단계를 잘 나누는 법

    여러 단계를 매끄럽게 이으려면, 각 단계가 무엇을 받고 무엇을 내놓는지 분명해야 한다. 한 단계의 출력이 다음 단계의 입력과 맞아야 흐름이 끊기지 않는다.

    질문좋은 답의 예
    이 단계는 무엇을 입력으로 받는가?CSV 원본, DOCX 초안, 폴더 경로처럼 구체적으로 쓴다
    이 단계는 무엇을 출력하는가?정규화 CSV, 요약표, 보고서 초안처럼 다음 단계가 받을 수 있게 쓴다
    이 단계의 책임은 어디까지인가?수집 단계는 수집만, 분석 단계는 분석만 하도록 나눈다
    완료를 어떻게 확인하는가?행 수, 누락 항목, 형식처럼 확인 기준을 둔다

    5.3 여러 단계를 한 번에 요청하기

    단계를 다 나눴으면 하나의 요청문으로 묶는다. 각 단계에 어울리는 Skill이 떠오르면 그 이름을 함께 지명하고, 떠오르지 않으면 Codex가 고르게 둔다. 어느 쪽이든 사람이 분명히 해야 할 것은 '단계'와 '단계 사이의 연결'이다.

    ⌨ 따라 입력해보기

    아래 업무를 여러 단계를 잇는 하나의 흐름으로 처리해줘. 목표: <최종 목표> 입력: <파일 / 폴더 / 데이터> 단계: 1) <첫 단계> 2) <둘째 단계> 3) <셋째 단계> 각 단계의 결과를 다음 단계의 재료로 이어서 진행하고, 단계마다 무엇을 만들었는지 짧게 알려줘.

    5.4 실습 — 세 단계를 하나의 작업 흐름으로 잇기

    5.1~5.3에서 본 대로, 연결되는 것은 '단계'다. 큰 일을 단계로 쪼개고, 각 단계가 알맞은 Skill을 도구로 쓰게 한다. 이번 실습은 세 단계짜리 작은 작업 흐름을 직접 짜서 한 번에 돌려 본다.

    만들 것 — 사내 'AI 스터디 모임' 모집 안내문 한 장. 흐름은 아래처럼 짠다.

    단계하는 일 (쓰는 Skill)받는 것내놓는 것
    1색·폰트·톤 정하기 — taste-skill안내문 주제브랜드 킷
    2헤더 이미지 만들기 — imagegen1단계의 톤·분위기헤더 이미지
    3디자인·이미지·문구 합쳐 문서로 — docx1·2단계 결과 + 안내 문구안내문 Word

    2단계는 1단계가 끝나야, 3단계는 1·2단계가 끝나야 시작된다. 단계가 이렇게 이어지는 것이 '흐름'이다. 각 단계 옆의 Skill은 그 단계가 집어 쓰는 도구일 뿐, 서로 연결된 것이 아니다.

    흐름을 다 짰으면 단계 순서대로 한 번에 요청한다. 각 단계에 어울리는 Skill을 함께 지명하면, 어떤 Skill이 무슨 일을 했는지 눈으로 확인할 수 있다.

    ⌨ 따라 입력해보기 · 작업 흐름 실행

    사내 'AI 스터디 모임' 모집 안내문 한 장을 만들고 싶어. 아래 세 단계를 하나의 흐름으로 이어서 진행해줘. ① taste-skill로 'AI 스터디 모임' 안내문에 어울리는 색·폰트·톤을 정해줘 → ② imagegen으로 ①에서 정한 톤에 맞는 헤더 이미지를 만들어줘 — 차분한 스터디 분위기, 글자는 넣지 마 → ③ docx Skill로 ①의 색·폰트와 ②의 이미지를 적용해, 모임 이름·매주 수요일 저녁 7시·신청 방법(사내 메일)을 담은 A4 한 장 안내문을 만들어줘. 각 단계의 결과를 다음 단계의 재료로 넘기고, 단계마다 어떤 Skill을 썼는지 알려줘.

    👁 관찰 포인트

    세 단계가 차례로 손을 거쳤다 — 1단계가 정한 색·폰트가 3단계 문서에 입혀지고, 2단계가 만든 이미지가 3단계 문서에 박혔다. 우리가 이은 것은 '단계'다. taste-skill·imagegen·docx는 각 단계가 필요해서 집어 든 도구일 뿐, 서로 연결된 것이 아니다. 한 단계의 결과물이 다음 단계의 재료가 되는 것 — 이것이 5장이 말하는 '연결'이고 '흐름'이다.

    📄 예시 산출물

    이 자료로 사용자가 얻을 수 있는 AI 동향과 인사이트를 분석 보고서 형태로 만들어줘. 더 깊은 분석을 위해 다음을 지켜줘. 1) 세 데이터를 서로 연결지어 발견할 수 있는 것을 3가지 제안해줘. (예: 'Ollama 인기 모델이 Arena에서도 상위인가' 같은 교차 분석) 2) 각 데이터에서 통계적으로 튀는 값(이상치, 변곡점, 패턴 변화)을 찾아줘. 3) 초보자도 이해할 수 있도록 쉽게 설명하고 인사이트를 제공해줘. 4) 단순 통계 나열에 그치지 말고, '그래서 뭐?'에 답하는 내용이 반드시 있어야 해.

    6.2~6.5에서 수집·분석·시각화·보고서를 하나씩 해봤다. 자동화는 이 네 단계를 사람 손에서 떼어내는 일이다. 한 번에 되는 것이 아니라, 세 걸음을 차례로 거친다.

    1걸음 · 묶기 — 네 단계를 한 번의 요청으로

    6.2~6.5는 한 채팅에서 이어가며 한 단계씩 해봤다. 이번엔 그 네 단계를 한 번의 요청으로 잇는다. 새 채팅창을 열어 아래처럼 입력해도 되는데, 한 가지 요령이 있다.

    ⌨ 따라 입력해보기 · 묶기

    AI 모델 동향을 정리하고 싶어. 아래 네 단계를 하나의 흐름으로 이어서 진행해줘. ① 앞서 만든 'ai-data-downloader' Skill로 세 사이트 데이터를 수집하고 → ② 교차 분석해 인사이트를 뽑고 → ③ 차트로 시각화하고 → ④ Word 보고서로 정리해줘. 각 단계의 결과를 다음 단계의 재료로 넘겨줘. 보고서의 분석 제목, 섹션 제목, 결론, '그래서 뭐?'는 하드코딩하지 말고 매 실행의 데이터와 이전 실행 비교 결과로 새로 작성해줘. 레이아웃은 재사용해도 되지만, 인사이트 문장은 재사용하지 마. 변화가 작아도 지표별 변화표를 포함해줘.

    🔗 왜 Skill 이름을 콕 집어 불렀나

    새 채팅창에는 앞 대화 내용이 남아 있지 않다. 그냥 '세 사이트에서 수집해줘'라고만 하면 Codex는 어느 사이트인지 알 수 없다.

    하지만 Skill은 채팅창이 바뀌어도 남는다. 6.2에서 저장해 둔 'ai-data-downloader' Skill을 이름으로 부르면, 새 채팅에서도 세 사이트를 정확히 안다.

    1장에서 배운 '프롬프트는 대화가 끝나면 사라지고, Skill은 남는다'가 바로 이 장면이다.

    💡 팁

    묶을 때 두 가지 옵션을 더할 수 있다. 분기 — '어제와 비교해 1위가 바뀐 부문이 있을 때만 보고서에 주요 변동 섹션을 넣어줘'처럼 조건에 따라 다르게. 병렬 — '세 사이트 수집을 동시에 진행해줘'처럼 여러 일을 한꺼번에. 둘 다 묶음을 더 똑똑하게·빠르게 만드는 선택지일 뿐, 그 자체가 자동화는 아니다.

    2걸음 · 굳히기 — 그 흐름을 Skill로 저장

    6.2에서 수집 세 동작을 'ai-data-downloader' Skill로 저장했다. 이번엔 방금 묶은 전체 흐름(수집→분석→시각화→보고서)을 하나의 Skill로 저장한다.

    ⌨ 따라 입력해보기 · 굳히기

    방금 진행한 전체 흐름을 'ai-동향-리포트'라는 Skill로 저장해줘. Skill에는 보고서 템플릿만 재사용하고, 분석 질문·인사이트·결론은 매 실행마다 수집 데이터에서 새로 만들도록 구성해줘.

    👁 관찰 포인트

    이제 그 긴 흐름이 한 줄 명령이 됐다. "ai-동향-리포트 Skill 실행해줘" 한 줄이면 네 단계가 다 돈다. 하지만 아직은 '내가' 그 한 줄을 직접 쳐야 한다 — 마지막 걸음이 남았다.

    3걸음 · 예약 — 사람 없이 정해진 때 스스로 실행

    마지막 걸음. 굳혀 둔 Skill을 '정해진 때에 사람 없이' 실행되게 예약한다. 이 걸음을 거쳐야 비로소 내가 채팅창을 열지 않아도 일이 끝나 있다.

    ⌨ 따라 입력해보기 · 예약

    'ai-동향-리포트' Skill을 매주 월요일 오전 9시에 자동으로 실행하고, 완성된 보고서를 나에게 알려줘. 지난주와 비교해 큰 변화가 있을 때만 알림을 보내줘.

    ⚠ 주의

    예약 실행(스케줄) 기능의 정확한 이름과 사용법은 Codex App 버전에 따라 다를 수 있다.

    ★ 핵심 문장

    자동화의 핵심은 '분기'나 '병렬'이 아니라 '사람이 빠지는 것'이다. 묶기는 손을 덜고, 굳히기는 한 줄로 줄이고, 예약은 손을 아예 뗀다. 세 걸음을 다 거쳐야 '자동화했다'고 말할 수 있다.

    CHAPTER 7

    7장. Skill을 독립 앱으로

    바이브 코딩으로 Codex App 밖에서도 도는 앱 만들기

    🎯 이 장에서 배울 것
    • 프롬프트 → Skill → 독립 앱으로 이어지는 사다리
    • 바이브 코딩 — 코드는 Codex가 쓰고, 사람은 무엇을 만들지만 말한다
    • 독립을 얻을 때 무엇을 내주는가 (트레이드오프)

    7.1 Skill에서 독립 앱으로

    6장에서 만든 'ai-동향-리포트' Skill은 강력하지만 두 가지에 묶여 있다 — Codex App이 있어야 돌고, 분석할 때마다 클라우드 AI를 부른다. 한 걸음 더 나아가면, 그 일을 어디에도 기대지 않는 독립 앱으로 만들 수 있다.

    프롬프트
    그때만
    →
    Skill
    반복 가능, 단 Codex App 안
    →
    독립 앱
    Codex App도 클라우드도 없이 혼자

    코드는 Codex가 쓰고, 사람은 무엇을 만들지만 말한다 — 이것이 '바이브 코딩'이다. 방법은 지금까지와 똑같다. 한국어로 부탁하면 된다.

    7.2 실습 — Skill을 독립 앱으로 만들기

    ⌨ 따라 입력해보기

    'ai-동향-리포트' Skill을 확인하고, 같은 일을 하는 독립 앱으로 만들어줘. Codex App과 클라우드 AI 없이 — 내 컴퓨터에서 Python과 Ollama(로컬 AI 모델)만으로 — 돌아가게 구현해줘.

    이 한 줄이면 Codex가 수 분 안에 앱을 만든다 — 데이터 수집·분석·차트·보고서를 Python으로 다시 짜고, 분석에 로컬 모델을 쓰도록 연결해서. 강사가 강의 중에 라이브로 시켜 보여줘도 좋다.

    다만 독립은 공짜가 아니다. Codex App Skill과 독립 앱은 이렇게 갈린다.

    항목Codex App Skill (6장)독립 앱 (7장)
    AI 모델클라우드 프런티어 모델내 PC의 로컬 모델 (Ollama)
    분석 품질높다낮아진다
    필요한 것Codex AppPython·Ollama 설치
    Ollama 없이 돌리면—AI 분석이 빠지고 규칙 기반으로 대체
    독립성Codex App·인터넷·계정 필요인터넷·계정·비용 없이 혼자 돎
    ⚠ 주의

    독립 앱을 실제로 돌리려면 Python과 Ollama를 설치해야 한다. 코드 경험이 없으면 이 설치가 문턱이 될 수 있다. 강의에서는 강사 시연으로 보고, 관심 있는 분이 각자 해보는 것을 권한다.

    ★ 핵심 문장

    독립을 얻는 대신 성능을 내준다 — 공짜가 아니다. 흥미롭게도 이것은 6장에서 데이터로 본 그대로다. 로컬 오픈 모델은 클라우드 프런티어 모델을 아직 따라잡지 못한다. 7장의 앱을 직접 돌려 보면, 그 격차를 글이 아니라 결과물로 체감하게 된다.
